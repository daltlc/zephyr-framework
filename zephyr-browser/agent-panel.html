Welcome to Zephyr Browser

This is a proof-of-concept for something we're calling WebMCP. The basic idea: what if the browser itself was the AI agent, instead of just being the window you look through?

What is WebMCP?

Right now, if you want an AI agent to control a webpage, you either need a server sitting in the middle or you embed the agent directly in the page (which means your API keys are just... sitting there in the source code). Neither is great.

WebMCP is a different approach. The browser's own process becomes the agent. It holds your API keys, talks to the AI model, and sends commands to the webpage through a secure bridge. The webpage just says what it can do. It never sees your keys.

How is this different?

Server Agent

  • Keys on your server
  • Agent runs remotely
  • Needs WebSocket bridge
  • Server infrastructure required

Embedded Widget

  • Keys in the page (XSS risk)
  • Agent runs in the page
  • No server needed
  • Visible in page source

WebMCP (this demo)

  • Keys in browser process
  • Agent runs outside page
  • No server needed
  • Page never sees keys

How does it work?

You type a message | v Agent Panel (this sidebar) | IPC v Main Process [API key lives here] | fetch() v Anthropic API returns tool calls | v Main Process forwards tool call via IPC | v Preload Bridge (the "WebMCP" layer) | v Zephyr.agent.act() on the web page | v Result flows back up, keys never crossed

Where could this go?

Right now we're using Electron's IPC as the bridge, but imagine if browsers had this built in. Something like navigator.webmcp where any webpage could declare its capabilities and any agent, whether it's built into the browser, an extension, or a desktop app, could discover and use them. No custom bridges, no exposed keys.

This demo is us proving the pattern works. The standard part comes later.
